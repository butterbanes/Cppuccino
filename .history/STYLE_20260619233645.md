# STYLE GUIDE

This project implements Java-style wrapper types in modern C++ while following a modified version of the Google C++ Style Guide.

The goal is to maintain consistency, readability, and a Java-like API surface while still writing idiomatic C++.

---

## Design Philosophy

This library is designed to:

- Emulate Java wrapper classes (`Integer`, `Double`, `Boolean`, etc.)
- Provide a familiar Java-like API (`toString()`, `equals()`, `intValue()`)
- Remain efficient and idiomatic in C++
- Favor clarity over extreme performance micro-optimizations

---

## Language Standard

- Minimum standard: C++20
- Prefer modern features:
  - `std::string`
  - `std::optional`
  - `constexpr`
  - `nullptr`

---

## Naming Conventions

### Classes (Java-style)

```cpp
class Boolean;
class Byte;
class Character;
class Double;
class Float;
class Integer;
class Long;
class Short;
```

- Use PascalCase
- Match Java class names where reasonable

---

### Methods (Java-style)

```cpp
int intValue() const;
std::string toString() const;
bool equals(const Integer& other) const;
```

- Use camelCase
- Follow Java naming conventions (not snake_case)

---

### Variables

```cpp
int value;
bool isValid;
```

- Use camelCase
- Be descriptive and avoid abbreviations

---

### Constants

```cpp
static constexpr int MAX_VALUE = 100;
```

- Use UPPER_CASE
- Prefer constexpr over macros

---

## Formatting

Formatting is enforced via clang-format (Google-based).

Key rules:

- 4 spaces indentation
- Column limit: 100
- Braces on same line

Example:
```cpp
if (value > 0) {
    return true;
}
```

---

## C++ Best Practices

### Const correctness

Always mark methods const when possible:

```cpp
std::string toString() const;
```

---

### Pass by reference

Avoid unnecessary copies:

```cpp
bool equals(const Integer& other) const;
```

---

### Avoid raw ownership

- Prefer references
- Use smart pointers when ownership is required

---

## API Design Rules

### 1. Java-like behavior takes priority

Wrapper types should behave like Java equivalents:

```cpp
Integer a(10);
Integer b(10);

a.equals(b);
a == b;
```

---

### 2. Provide explicit conversion methods

Do not rely only on implicit conversions:

```cpp
int intValue() const;
double doubleValue() const;
```

---

### 3. Keep objects simple and predictable

- Prefer immutability where possible
- Avoid hidden state changes

---

### 4. Operator overloading (optional, consistent only)

If used, must match Java methods:

```cpp
bool operator==(const Integer& other) const;
```

Must always be consistent with:

```cpp
bool equals(const Integer& other) const;
```

- Headers define interface
- Source files implement logic (unless header-only)

---

## Testing

All new features should include tests.

Recommended framework:

- GoogleTest

Test requirements:

- Constructors
- Equality
- Conversion methods
- Edge cases

---

## Prohibited Patterns

Avoid:

- Mixing naming styles (to_string() vs toString())
- Global variables
- Macro-heavy code
- Platform-specific logic without abstraction
- Overly complex template metaprogramming

---

## Contribution Consistency Rule

If unsure:

Follow existing code style in the module first, then this guide second, then Google C++ style last.

Consistency > perfection.

---

## Summary

This project intentionally blends:

- Java-like API design
- Modern C++ implementation
- Google C++ style formatting baseline

The guiding principle is:

“Familiar to Java developers, idiomatic to C++ developers.”
